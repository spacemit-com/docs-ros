# Agent

## 1. 模块概述

`omni_agent` 是 SDK 中的端到端语音对话应用，位于 `application/native/omni_agent`。它把 audio、VAD、ASR、LLM、TTS、声纹和 MCP 工具调用组合成完整语音助手：

```text
AudioCapture -> VAD -> 声纹验证（可选） -> ASR -> LLM/MCP -> TTS -> AudioPlayer
```

### 1.1 整体架构图

下图覆盖标准 `voice_chat` 进程，不包含 `voice_chat_aec` 软件 AEC 模式。

![voice_chat 整体架构图](assets/voice_chat_architecture.svg)

当前主推标准 `voice_chat` 入口，默认构建不启用 `voice_chat_aec`。SDK 配套的音频子板提供硬件侧音频前处理能力，包括回声消除、降噪和 360° 声源方位估计（DOA，用于判断说话人方向）。常规语音对话默认使用硬件侧音频处理，不需要软件回采 AEC。

当前提供两个入口：

| 程序 | 源码入口 | 适用场景 |
| --- | --- | --- |
| `voice_chat` | `application/native/omni_agent/examples/voice_chat.cpp` | 标准模式，默认构建和主推入口，适合配套音频子板或外设自带音频处理的场景。 |
| `voice_chat_aec` | `application/native/omni_agent/examples/voice_chat_aec.cpp` | 可选软件回采 AEC 模式，启用 WebRTC AEC3，全双工 48kHz 音频链路，默认不编译。 |

主要依赖组件：

| 组件 | 路径 | 作用 |
| --- | --- | --- |
| audio | `components/multimedia/audio` | 音频采集、播放和重采样。 |
| VAD | `components/model_zoo/vad` | Silero VAD，检测说话开始、结束和 barge-in。 |
| ASR | `components/model_zoo/asr` | SenseVoice 等语音识别后端。 |
| TTS | `components/model_zoo/tts` | Matcha-TTS / Kokoro 语音合成。 |
| voiceprint | `components/model_zoo/voiceprint` | 可选说话人验证。 |
| MCP | `components/agent_tools/mcp` | 可选工具调用客户端。 |

## 2. 环境准备

### 前置条件

SDK 源码获取和基础编译环境配置统一参考 [2.3-构建编译](../02-快速入门/2.3-构建编译.md)。完成 SDK 初始化后，回到本文继续执行“构建编译”。

后续命令默认在 `spacemit_robot` SDK 根目录执行。

### 构建编译

Agent 使用 `target/k3-com260-omni-agent.json` 目标配置。首次构建建议在 SDK 根目录执行完整构建，确保 audio、VAD、ASR、TTS、声纹和 MCP 依赖都已生成：

```bash
source build/envsetup.sh
lunch k3-com260-omni-agent
m
```

只改 Agent 代码时，可以只编 `omni_agent`：

```bash
source build/envsetup.sh
lunch k3-com260-omni-agent
cd application/native/omni_agent
mm
```

默认只构建标准 `voice_chat`。如确实需要软件回采 AEC，再显式启用 `voice_chat_aec`：

```bash
cd application/native/omni_agent
mm -DUSE_AEC=ON
```

构建完成后，`voice_chat` 会安装到 `output/staging/bin`；显式启用 `USE_AEC=ON` 时，`voice_chat_aec` 也会安装到同一目录。加载 `build/envsetup.sh` 后可直接运行。

常用 CMake 选项：

| 选项 | 默认值 | 说明 |
| --- | --- | --- |
| `USE_MCP` | `ON` | 启用 MCP 工具调用支持。 |
| `USE_AEC` | `OFF` | 可选构建 WebRTC AEC3 软件回采版本 `voice_chat_aec`，默认关闭。 |

运行前还需要一个 OpenAI 兼容 LLM 服务。先安装 `llama-server` 工具包，并把 LLM 模型放到 `~/.cache/models/llm`：

```bash
sudo apt install llama.cpp-tools-spacemit

mkdir -p ~/.cache/models/llm
cd ~/.cache/models/llm
wget https://archive.spacemit.com/spacemit-ai/model_zoo/llm/Qwen3-0.6B-Q4_0.gguf
```

然后启动本地模型服务：

```bash
llama-server -m ~/.cache/models/llm/Qwen3-0.6B-Q4_0.gguf \
    -t 4 --reasoning-budget 0 --port 9191 --host 0.0.0.0
```

## 3. 示例使用

### 3.1 标准语音对话

```bash
voice_chat --llm-url http://localhost:9191 --tts matcha:zh-en
```

关键日志包括：

```text
[1/5] LLM 后端: http://localhost:9191 OK
[2/5] 初始化 VAD... OK (Silero VAD)
[3/5] 初始化 ASR... OK
[4/5] 初始化 TTS (matcha:zh-en)... OK
[5/5] 初始化音频设备... OK
[等待语音输入...]
```

常用参数：

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `--llm-url <url>` | 必填 | OpenAI 兼容 LLM 服务地址。 |
| `--model <name>` | `qwen2.5:0.5b` | LLM 模型名。 |
| `--tts <engine>` | `matcha:zh` | `matcha:zh`、`matcha:en`、`matcha:zh-en`、`kokoro` 或 `kokoro:<voice>`。 |
| `-i, --input-device <id>` | 系统默认 | 录音设备索引。 |
| `-o, --output-device <id>` | 系统默认 | 播放设备索引。 |
| `-l, --list-devices` | 无 | 列出音频设备。 |
| `--capture-rate <hz>` | 16000 | 标准模式录音采样率。 |
| `--playback-rate <hz>` | 48000 | 播放采样率。 |
| `--mcp-config <path>` | 空 | MCP 配置文件。 |

### 3.2 启用声纹验证

先注册说话人：

```bash
register_speaker -n muggle -t 2
```

再启动语音对话并要求只接受 `muggle`：

```bash
voice_chat --llm-url http://localhost:9191 --tts matcha:zh-en \
    -vp --vp-database ./speakers.db --vp-threads 2 --vp-verify muggle
```

实测日志中，声纹通过时继续 ASR；不通过时丢弃音频：

```text
[VP] 验证 "muggle": 通过 (score: 0.717)
[ASR] 开始识别...
[VP] 验证 "muggle": 不通过 (score: 0.546)
[VP] 未识别说话人，丢弃音频
```

### 3.3 用户打断 barge-in

TTS 播放期间，VAD 会继续监听麦克风。当连续多帧检测到用户说话时，系统停止当前播放并中断 LLM 生成：

```text
[Barge-in] 用户打断 (连续5帧, prob=0.856)，停止播放
[LLM] 已因 barge-in 中断生成
[TTS] Barge-in 打断，保留音频缓冲区
```

标准 `voice_chat` 是默认推荐路径。配套音频子板已提供回声消除、降噪和 360° 声源方位估计（DOA）等硬件侧前处理；只有在没有硬件侧音频处理、且确实需要软件回采 AEC 时，才启用 `voice_chat_aec`。构建 `-DUSE_AEC=ON` 后运行：

```bash
voice_chat_aec --llm-url http://localhost:9191 --tts matcha:zh-en
```

`voice_chat_aec` 使用 `--sample-rate` 配置全双工采样率，不使用标准模式的 `--capture-rate` / `--playback-rate`。

### 3.4 MCP 工具调用

MCP 组件路径为 `components/agent_tools/mcp`，示例配置位于 `components/agent_tools/mcp/examples/configs/`：

| 配置 | 说明 |
| --- | --- |
| `config_stdio.json` | 通过 stdio 自动启动 Calculator 和 TimeService。 |
| `config_socket.json` | 连接 Unix Socket 服务。 |
| `config_http.json` | 连接 HTTP MCP 服务。 |
| `config_mixed.json` | 混合 stdio 与 socket。 |
| `config_registry.json` | 通过注册中心动态发现 HTTP 服务。 |

如果只连接已经部署好的远端 MCP 服务，`voice_chat` 侧只需要传入 `--mcp-config`。如果要运行仓库自带的 Python 示例服务，需要先创建并激活虚拟环境，再安装 MCP 服务端依赖：

```bash
sudo apt install python3-venv python-is-python3 python3-pip
python3 -m venv ~/.comm-env
source ~/.comm-env/bin/activate

python -m pip install flask mcp starlette uvicorn psutil \
    --prefer-binary \
    --retries 0 \
    --timeout 2 \
    --index-url https://git.spacemit.com/api/v4/projects/33/packages/pypi/simple \
    --extra-index-url https://mirrors.aliyun.com/pypi/simple/
```

这里使用 SpaceMIT PyPI 源作为主源，是为了获取 RISC-V 可用的原生依赖 wheel，例如 `pydantic-core`；同时用公开镜像作为补充源获取 `py3-none-any` 纯 Python 包。`--prefer-binary` 用于避免 pip 优先选择公开源上的新版源码包并触发本地编译。SpaceMIT 源没有的包可能会触发 GitLab PyPI 上游转发到 `pypi.org`，`--retries 0 --timeout 2` 用于快速跳过转发超时等待。

MCP 配置中的服务连接方式由 `type` 字段表示：

```json
{
  "servers": [
    {
      "name": "Calculator",
      "type": "stdio",
      "command": "python3",
      "args": ["examples/services/calculator/stdio_server.py"]
    },
    {
      "name": "SystemMonitor",
      "type": "socket",
      "path": "/tmp/mcp_system_monitor.sock"
    },
    {
      "name": "TimeService",
      "type": "http",
      "url": "http://localhost:8002/mcp"
    }
  ]
}
```

运行示例：

```bash
source ~/.comm-env/bin/activate
cd components/agent_tools/mcp/examples
./start_all_services.sh start

cd -
voice_chat --llm-url http://localhost:9191 \
    --tts matcha:zh-en \
    --mcp-config components/agent_tools/mcp/examples/configs/config_registry.json
```

MCP 场景建议显式使用 `--tts matcha:zh-en`。工具返回内容里经常包含英文、数字、系统字段名或路径，默认 `matcha:zh` 更适合纯中文文本，不适合朗读这类中英混合结果。

## 4. 应用开发

Agent 是组合应用，没有独立的 "AgentService" 入口类。本章面向两类应用开发者：

- **想接入 MCP 工具**：直接复用 `components/agent_tools/mcp` 提供的 `mcp::MCPClient` / `mcp::MCPManager` SDK，把工具调用接到自己的 LLM 应用里——这是 Agent 模块对外**唯一稳定可复用的 SDK**。
- **想自建语音 Agent**：把 audio / VAD / ASR / LLM / TTS 五个组件按 omni_agent 的模式组装起来。本节给出最小骨架；上游各组件 API 详见各自 `4.x §4`。

**MCP 的 C++ 命名空间是 `mcp::`**，与其它 SpaceMIT 组件（`SpacemiT::` / `SpacemitAudio::`）不同，应用代码中 `using namespace` 时需要显式区分。

`omni_agent` 内部公共代码（`engine_init` / `voice_pipeline` / `voice_common` / `aec_duplex_processor` / `mcp_helper`）属于应用层胶水，不保证 ABI 稳定，也不在本节展开 API 表。下表仅供需要修改 omni_agent 内部的开发者参考：

| 路径 | 说明 |
| --- | --- |
| `include/engine_init.hpp` / `src/engine_init.cpp` | 初始化 LLM、VAD、ASR、TTS、MCP、声纹。 |
| `include/voice_pipeline.hpp` / `src/voice_pipeline.cpp` | ASR → LLM → TTS 的对话流水线。 |
| `include/voice_common.hpp` / `src/voice_common.cpp` | TTS 后端解析、全局状态和公共工具函数。 |
| `include/aec_duplex_processor.hpp` / `src/aec_duplex_processor.cpp` | AEC 全双工音频处理。 |
| `include/mcp_helper.hpp` / `src/mcp_helper.cpp` | MCP 配置解析和工具格式转换。 |

新增业务逻辑时，优先在 `voice_pipeline` 周围扩展，不要把组件初始化、音频设备控制和 LLM/TTS 生成逻辑混在同一个函数里。新增 TTS 后端、ASR 后端或声纹策略时，应先在对应组件内完成封装，再由 `engine_init` 读取配置。

### 4.1 MCP SDK 接口

MCP（Model Context Protocol）SDK 把外部工具服务（计算器、时间、系统监控等）抽象成统一的 `Tool` / `ToolResult`，让 LLM 通过 Tool Calling 调用。完整接口以 `components/agent_tools/mcp/include/mcp_service.hpp` 为准。

#### 4.1.1 常用数据结构

| 类型 | 说明 |
| --- | --- |
| `Tool` | 工具定义。字段 `name`、`description`、`inputSchema`（`nlohmann::json`，OpenAI tools 格式）。提供 `fromJson(j)` / `toJson()`。 |
| `ToolResult` | 工具调用结果。字段 `success`、`contents`（文本结果列表）、`error`（失败时的错误描述）、`rawResult`（原始 JSON）。 |
| `ServerInfo` | 服务器握手信息：`name`、`version`、`capabilities`（JSON）。 |
| `ServerState` | 服务器状态机：`Disconnected` / `Connecting` / `Initializing` / `Ready` / `Reconnecting` / `Error`。 |
| `ServerStatus` | 单服务器运行时快照：`name`、`state`、`tools`、`lastError`。 |
| `ErrorCode` | 错误码枚举。常见值：`ConnectionFailed=101`、`Timeout=105`、`InitializeFailed=202`、`ToolNotFound=301`、`ToolCallFailed=303`。 |
| `MCPException` | SDK 异常类型。`exception.code()` 返回 `ErrorCode`。 |

#### 4.1.2 Transport 抽象

| 接口 | 说明 | 参数 | 返回值 |
| --- | --- | --- | --- |
| `Transport`（抽象类） | `connect / disconnect / isConnected / send / receive(timeout) / typeName`。一般不直接实现，用工厂函数。 | — | — |
| `createStdioTransport(StdioConfig)` | 通过 stdio 启动子进程并通信。`StdioConfig` 字段：`command`、`args`、`workingDir`、`env`、`startupTimeout`、`requestTimeout`。 | `StdioConfig`。 | `unique_ptr<Transport>`。 |
| `createUnixSocketTransport(UnixSocketConfig)` | 连接已有 Unix 套接字。`UnixSocketConfig` 字段：`socketPath`。 | `UnixSocketConfig`。 | `unique_ptr<Transport>`。 |
| `createHttpTransport(HttpConfig)` | 通过 HTTP（含 SSE 流式）连接。`HttpConfig` 字段：`url`、`headers`、`timeout`、`enableSSE`。 | `HttpConfig`。 | `unique_ptr<Transport>`。 |

#### 4.1.3 单服务器客户端 `MCPClient`

| 接口 | 说明 | 参数 | 返回值 |
| --- | --- | --- | --- |
| `MCPClient(unique_ptr<Transport>, ClientConfig)` | 用一个 Transport 构造客户端。`ClientConfig` 字段：`name`、`version`、`requestTimeout`。 | Transport + 配置。 | 实例。 |
| `connect()` | 建立 Transport 连接。 | 无。 | `bool`。 |
| `initialize(timeout=0)` | 完成 MCP 握手（initialize / initialized / tools/list）。`timeout=0` 用配置默认。 | 超时。 | `bool`。 |
| `listTools(timeout=0)` | 获取最新工具列表，必要时刷新。 | 超时。 | `vector<Tool>`。 |
| `callTool(name, args, timeout=0)` | 调用指定工具，`args` 是 OpenAI tools 参数 JSON。 | 名字、参数、超时。 | `ToolResult`。 |
| `onToolsChanged(handler)` | 注册"工具列表变更"通知回调（服务器主动推送时触发）。 | `function<void()>`。 | 无。 |
| `shutdown()` | 关闭连接，释放 Transport。 | 无。 | 无。 |
| `isConnected()` / `isInitialized()` / `serverInfo()` / `name()` | 状态查询。 | 无。 | 各自类型。 |

#### 4.1.4 多服务器管理 `MCPManager`

| 接口 | 说明 | 参数 | 返回值 |
| --- | --- | --- | --- |
| `addStdioServer(name, StdioConfig)` / `addUnixSocketServer(name, UnixSocketConfig)` / `addHttpServer(name, HttpConfig)` | 注册不同类型的服务器，**未自动启动**。 | 名字 + 对应配置。 | 无。 |
| `removeServer(name)` | 停止并移除服务器。 | 名字。 | `bool`。 |
| `startServer(name)` / `stopServer(name)` / `startAll()` / `stopAll()` | 单独或批量控制服务器启停。 | 名字（或全部）。 | `bool` / 无。 |
| `waitForAnyServer(timeout=30s)` / `waitForAllServers(timeout=60s)` | 阻塞等待至少一个 / 全部服务器进入 `Ready`。 | 超时。 | `bool`。 |
| `getAllTools()` / `getAllToolsJson()` | 跨所有就绪服务器收集工具列表。`getAllToolsJson()` 直接返回 OpenAI tools 格式 JSON。 | 无。 | `vector<Tool>` / `json`。 |
| `callTool(name, args)` | 在所有服务器中找到名为 `name` 的工具并调用。 | 工具名 + 参数。 | `ToolResult`。 |
| `findServerForTool(name)` | 查询某工具属于哪个服务器。 | 工具名。 | `string`。 |
| `getStatuses()` / `getStatus(name)` / `readyServerCount()` / `hasAvailableServers()` | 状态查询与统计。 | 无 / 名字。 | 各自类型。 |
| `onServerEvent(handler)` / `onToolChange(handler)` | 注册服务器状态变更和工具列表变更回调。 | 回调函数。 | 无。 |

#### 4.1.5 MCP 配置 JSON 字段

`omni_agent --mcp-config` 接受的 JSON 配置字段（与 `mcp/examples/configs/*.json` 对应）：

| 字段 | 类型 | 含义 | 默认值 | 必填 |
| --- | --- | --- | --- | --- |
| `backend` | string | LLM 后端类型，`llama` 或 `openai`。 | `llama` | 否 |
| `url` | string | LLM API 地址，覆盖命令行 `--llm-url`。 | — | 是 |
| `model` | string | 模型名，覆盖命令行 `--model`。 | — | 是 |
| `timeout` | int | LLM 请求超时秒数。 | 120 | 否 |
| `system_prompt` | string | 系统提示词，覆盖默认值。 | 内置 | 否 |
| `servers` | array | MCP 服务列表，每条含 `name` + `type` + 类型特有字段（`command/args` for stdio, `path` for socket, `url` for http）。 | — | 是 |
| `registry_url` | string | 注册中心 URL，启用后会从注册中心动态发现 HTTP 服务。 | — | 否 |
| `registry_poll_interval` | int | 注册中心轮询间隔（秒）。 | 5 | 否 |

### 4.2 调用示例

MCP 组件编译后会把 `mcp_service.hpp` 和 `libmcp.so` 安装到 `output/staging`。下游组件链接 MCP 库的 CMake 写法：

```cmake
add_executable(my_app main.cpp)

find_library(MCP_LIB NAMES mcp
    PATHS ${CMAKE_INSTALL_PREFIX}/lib NO_DEFAULT_PATH)
find_path(MCP_INCLUDE_DIR NAMES mcp_service.hpp
    PATHS ${CMAKE_INSTALL_PREFIX}/include NO_DEFAULT_PATH)
if(NOT MCP_LIB OR NOT MCP_INCLUDE_DIR)
    message(FATAL_ERROR "mcp library not found. Build components/agent_tools/mcp first.")
endif()

target_include_directories(my_app PRIVATE ${MCP_INCLUDE_DIR})
target_link_libraries(my_app PRIVATE ${MCP_LIB})
```

依赖 `nlohmann_json`；`package.xml` 中声明 `<depend>mcp</depend>`。组合自建 Agent 还需链接 LLM / ASR / TTS / VAD 各组件库（参考各自 §4.2 的 CMake 块）。

#### 4.2.1 单 Server（stdio）

适用场景：本地起一个 MCP server 子进程（Python / 任何语言），调用其工具；最适合开发调试和单一工具集成。

调用步骤：

1. 用 RAII `SDKInit` 初始化 SDK。
2. 用 `StdioConfig` 描述子进程启动方式，传给 `createStdioTransport()`。
3. 构造 `MCPClient`，调 `connect()` → `initialize()`。
4. `listTools()` 看可用工具，`callTool(name, args)` 调用；最后 `shutdown()`。

```cpp
#include <iostream>
#include <mcp_service.hpp>

int main() {
    mcp::SDKInit sdk;

    mcp::StdioConfig cfg;
    cfg.command = "python3";
    cfg.args = {"examples/services/calculator/stdio_server.py"};

    mcp::MCPClient client(mcp::createStdioTransport(cfg),
                          mcp::ClientConfig{"my-client"});

    if (!client.connect() || !client.initialize()) {
        std::cerr << "MCP 连接失败" << std::endl;
        return 1;
    }

    for (const auto& tool : client.listTools()) {
        std::cout << "  - " << tool.name << ": " << tool.description << std::endl;
    }

    auto r = client.callTool("add", {{"a", 10}, {"b", 20}});
    if (r.success) std::cout << "结果: " << r.contents[0] << std::endl;
    else           std::cerr << "错误: " << r.error << std::endl;

    client.shutdown();
    return 0;
}
```

完整版见 `components/agent_tools/mcp/examples/simple_client.cpp`。

#### 4.2.2 多 Server + 工具变更回调

适用场景：聚合多个外部工具源（计算器、时间、系统监控、机器人控制），让 LLM 通过统一接口选择调用；典型用于 omni_agent 的 `--mcp-config` 模式。

调用步骤：

1. `MCPManager manager;`，逐一 `addStdioServer / addHttpServer` 添加服务器（**不会立刻启动**）。
2. `onServerEvent` / `onToolChange` 注册状态/工具变更回调（注册中心动态发现场景必需）。
3. `startAll()` 启动所有服务器；`waitForAnyServer()` 等到至少一个就绪。
4. `getAllTools()` 把跨服务器工具列表交给 LLM；LLM 决定调用哪个工具时，`manager.callTool(name, args)` 自动路由。

```cpp
#include <iostream>
#include <mcp_service.hpp>

int main() {
    mcp::SDKInit sdk;
    mcp::MCPManager manager;

    manager.addStdioServer("calculator",
                            {"python3", {"examples/services/calculator/stdio_server.py"}});
    manager.addHttpServer("time",
                           {"http://127.0.0.1:8002/mcp", {}, std::chrono::milliseconds(30000), true});

    manager.onServerEvent([](const std::string& name, mcp::ServerState s) {
        std::cout << "[Event] " << name << " -> " << static_cast<int>(s) << std::endl;
    });
    manager.onToolChange([](const std::vector<mcp::Tool>& tools) {
        std::cout << "[ToolChange] 当前共 " << tools.size() << " 个工具" << std::endl;
    });

    manager.startAll();
    if (!manager.waitForAnyServer()) return 1;

    // 把工具列表喂给 LLM 的 tools_json
    auto tools_json = manager.getAllToolsJson();

    // ... LLM 决定调 "add" 工具 ...
    auto r = manager.callTool("add", {{"a", 1}, {"b", 2}});
    if (r.success) std::cout << r.contents[0] << std::endl;

    manager.stopAll();
    return 0;
}
```

完整版见 `components/agent_tools/mcp/examples/multi_server.cpp`。

#### 4.2.3 组合自建 Agent

适用场景：不复用 omni_agent 二进制，自己用基础组件拼最小语音对话——例如想去掉 TTS、换成 IM 推送，或者在嵌入式应用里只接 ASR + LLM。

下面是一个简化骨架，把 VAD → ASR → LLM(+MCP) → TTS 串起来；变量初始化和资源释放参考各组件 §4.2。**实际完整代码见 `application/native/omni_agent/src/voice_pipeline.cpp`**。

```cpp
#include <memory>
#include <vector>
#include "asr_service.h"
#include "tts_service.h"
#include "vad_service.h"
#include "llm_service.h"
#include "audio_base.hpp"
#include <mcp_service.hpp>

int main() {
    // 1) 各组件初始化（详见各自 §4）
    auto vad = std::make_shared<SpacemiT::VadEngine>(
        SpacemiT::VadConfig::Preset("silero"));
    auto asr = std::make_shared<SpacemiT::AsrEngine>(
        SpacemiT::AsrConfig::Preset("sensevoice"));
    auto tts = std::make_shared<SpacemiT::TtsEngine>(
        SpacemiT::TtsConfig::Preset("matcha_zh"));
    spacemit_llm::LLMService llm(
        "qwen2.5-0.5b", "http://127.0.0.1:8080/v1", "",
        "You are a helpful assistant.", 256);

    // 2) MCP 工具（可选，参考 §4.2.2）
    mcp::SDKInit sdk;
    mcp::MCPManager mcp_mgr;
    mcp_mgr.addStdioServer("calculator",
        {"python3", {"examples/services/calculator/stdio_server.py"}});
    mcp_mgr.startAll();
    mcp_mgr.waitForAnyServer();
    auto tools_json = mcp_mgr.getAllToolsJson().dump();

    // 3) 麦克风采集（详见 6.3.3-audio §4.2.2）
    SpacemitAudio::AudioCapture capture(-1);
    std::vector<int16_t> speech_buffer;
    bool in_speech = false;

    auto vad_cb = std::make_shared<EndpointCb>(  // 见 4.1.3-VAD §4.2.3
        [&](std::vector<int16_t> chunk) {
            // 4) ASR
            auto r = asr->Recognize(chunk, 16000);
            if (!r || r->IsEmpty()) return;
            std::string text = r->GetText();

            // 5) LLM + Tool Calling
            std::vector<spacemit_llm::ChatMessage> messages = {
                spacemit_llm::ChatMessage::User(text)
            };
            auto chat_r = llm.chat_stream(
                messages,
                [&](const std::string& chunk_text, bool, const std::string&) {
                    return true;  // 流式逐字处理
                },
                tools_json);

            // 工具调用回填
            if (chat_r.HasToolCalls()) {
                // 解析 chat_r.tool_calls_json，调 mcp_mgr.callTool(...)，
                // 把结果作为 ChatMessage::Tool() 追加到 messages 再次 chat()
            }

            // 6) TTS 播放
            tts->CallToFile(chat_r.content, "/tmp/reply.wav");
            SpacemitAudio::AudioPlayer player(-1);
            player.PlayFile("/tmp/reply.wav");
        });
    vad->SetCallback(vad_cb);
    vad->Start();

    // 7) 把音频送 VAD（双声道 16kHz，混成单声道送 VAD）
    capture.SetCallback([&](const uint8_t* data, size_t size) {
        // 转 float、混音、SendAudioFrame；同时 vad_cb->Append(int16) 见 4.1.3-VAD §4.2.3
    });
    capture.Start(16000, 2, 4096);

    // 8) 主循环（按业务退出条件控制）
    std::this_thread::sleep_for(std::chrono::minutes(5));

    // 9) 收尾
    capture.Stop();
    capture.Close();
    vad->Stop();
    mcp_mgr.stopAll();
    return 0;
}
```

链路里每个组件都可独立替换：把 `SenseVoice` 换成 `Qwen3-ASR`、把 `Matcha-TTS` 换成 `Kokoro`、把本地 LLM 换成云端 OpenAI 兼容服务等。完整工业级实现包含 barge-in、声纹验证、TTS 流式打断等高级特性，参考 omni_agent 源码。

## 5. 调试指南

调试 Agent 时建议按链路分段确认，先保证 LLM 服务和音频设备可用，再检查 VAD、ASR、TTS、声纹和 MCP：

- 启动前用 `voice_chat -l` 列出音频设备，并用 audio 文档中的录音、播放命令单独验证。
- LLM 问题先检查 `llama-server` 是否监听在 `--llm-url` 对应地址，再看 `voice_chat` 初始化日志。
- 语音链路问题按 VAD 开始/结束、ASR 识别文本、LLM 回复、TTS 播放顺序定位。
- MCP 问题先独立运行 `components/agent_tools/mcp/examples` 下的服务，再传入 `--mcp-config`。
- barge-in 误触发优先记录播放音量、麦克风距离和是否使用硬件音频前处理。

## 6. 常见问题

| 现象 | 可能原因 | 处理 |
| --- | --- | --- |
| 启动时报必须指定 LLM 地址 | 未传 `--llm-url` | 启动 `llama-server` 后传入正确 URL。 |
| 没有麦克风或扬声器声音 | 设备索引不对 | `voice_chat -l` 查看设备，再用 `-i` / `-o` 指定。 |
| ASR 前几秒延迟大 | SenseVoice 首次 warmup | 以 warmup 后的后续轮次评估交互时延。 |
| 播放时误触发 barge-in | 麦克风收到扬声器回声 | 优先使用配套音频子板的回声消除和降噪能力，或降低音量；没有硬件侧处理时再启用 `voice_chat_aec`。 |
| 注册人经常识别不到 | 声纹分数低于阈值，或注册音频不够干净清晰 | 适当降低 `--vp-threshold`（默认 0.6），或重新录制干净、清晰、音量稳定的声纹样本。 |
| MCP 工具不可用 | 服务未启动或配置路径错 | 先用 MCP 示例独立验证，再传 `--mcp-config`。 |
